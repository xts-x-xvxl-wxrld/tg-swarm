# Change Log

## 2026-05-13

- Wired the remaining schedule-authoring gap into the live runtime: the orchestrator can now author recurring schedule actions during normal operator turns, the runtime validates and persists `create` / `pause` / `resume` actions from a machine-readable schedule appendix, and schedule persistence now exposes the minimal lookup and status-update helpers needed for that path.
- Added `wiki/spec/managed-account-operations.md` to define the agentic-first operational loop for managed MTProto work as `prepare -> execute -> observe -> adapt`, aligned to operator direction, orchestrator coordination, specialist tactical autonomy, approval-aware execution, and durable campaign-memory feedback.
- Finished the next campaign-memory integration slice: strategy and account-planning specialists now ingest canonical campaign memory plus their own local working-memory files, campaign sync preserves specialist-owned `agents/*.md` notes instead of regenerating them from artifacts, and the active specialists now rewrite their own tactical memory deliberately after successful runs.
- Reduced the live runtime's dependence on workflow stages by making `PurposeBuiltOrchestrator` resolve specialist turns from the primary campaign work item first, backfilling stage-only sessions into compatibility work items when needed, and passing bounded work-item goals into specialist runs while keeping workflow snapshots as operator-facing compatibility views.
- Realigned `wiki/spec/account-capability.md` to match the original managed-account capability grouping explicitly: `presence`, `messaging`, `engagement`, `dialog`, `identity`, and `social`, with basic message send now living clearly under `messaging`.
- Tightened `wiki/spec/account-capability.md` so the account-focused MTProto surface explicitly includes basic outbound message send as a first-class consequential write alongside presence, read-state actions, reactions, identity changes, and other user-like account behaviors.
- Added `wiki/spec/account-capability.md` to define the narrower account-focused MTProto surface under the broader Telegram capability layer, including inventory/health, presence, read-state actions, lightweight engagement like reactions, identity management, approval expectations, and rollout guidance for making managed accounts behave more like real users over time.
- Completed the next campaign-memory and operations cut: added `telegram_app/campaign_memory/` for full workspace bootstrapping, compatibility artifact views, prompt-safe memory loading, and session hydration; expanded campaign workspaces to create the broader Markdown and agent-memory surface; shifted scheduled work to build campaign-native background context without requiring a preexisting session; and tightened orchestrator review routing so the primary open work item now outranks raw artifact presence when deciding whether to refresh work or enter review.
- Updated the scheduler-related runtime docs to match the current implementation: the app-runtime architecture spec and campaign-operations runtime plan now describe the dedicated `server.py --run-scheduler` worker, lease-protected due-schedule dispatch, scheduled specialist execution instead of placeholder-only work-item creation, simple schedule health thresholds with auto-pause, and the current latest-session bridge used while some artifacts still remain session-scoped.
- Completed the first operational cut of recurring scheduling: due schedules now execute specialist work instead of only creating placeholder items, schedules track outcome health and can auto-pause after repeated misses, session lookup now resolves campaign context for background runs, and `server.py --run-scheduler` starts a dedicated lease-protected scheduler worker so webhook, polling, and multi-instance app processes do not all dispatch the same due work by default.
- Landed the next campaign-runtime slices in code without a flag-day rewrite: added durable `telegram_app/work_items/` and `telegram_app/scheduling/` packages, introduced work-item and schedule runtime models, taught the orchestrator to track discovery/strategy/account-planning work items through creation, review, revision, completion, and schedule-triggered refresh, and now explicitly cancel legacy planning approvals when old `waiting_for_approval` state is released back into conversational review.
- Landed the first campaign-runtime cutover slice in code: added `telegram_app/campaigns/` for file-backed campaign workspace bootstrapping plus `campaign.json` metadata, gave sessions first-class `campaign_id` and `campaign_workspace_path` fields, taught `TelegramAppService` to auto-attach or backfill campaigns and sync the intake objective into campaign metadata, and updated the runtime context plus code index to expose the new campaign seam without removing the existing stage-driven flow yet.
- Synced the remaining older platform specs to the campaign-operations model: `wiki/spec/telegram-core-platform.md` now describes campaigns, delegated work, and recurring operating cadence instead of a session-only workflow story, and `wiki/spec/telegram-capability-layer.md` now explicitly supports campaign-attached work items, schedule-driven reads, and approval-aware write routing.

## 2026-05-12

- Tightened the campaign-operations and campaign-memory docs so the state model is unambiguous: campaigns are now the durable root object for downstream records, sessions are explicitly only the operator interaction layer, schedule-triggered work resolves campaign context directly, `active_session_id` and campaign-level stage/mode fields were removed from the recommended memory contract, and the runtime-refactor plan now includes an explicit migration phase to remove legacy planning approval state instead of letting it survive by accident.
- Added `wiki/plan/campaign-operations-runtime-refactor.md` and `wiki/plan/campaign-memory-implementation.md` as the first implementation plans directly subordinate to `wiki/spec/campaign-operations-model.md`: the runtime-refactor plan covers campaign-aware orchestration, work items, scheduling, and bounded specialist autonomy, while the memory plan covers campaign workspace layout, shared versus agent-local memory, lightweight metadata, and migration from the current artifact model.
- Updated the existing runtime, session, marketing-mode, and approval specs to align with the new campaign-operations model: sessions are now described as campaign-attached conversational threads, the runtime now targets campaign memory plus work-item and scheduling support, the marketing MVP is framed as an ongoing operating system rather than only a one-pass workflow, and approvals are now scoped more explicitly around consequential execution instead of normal planning or recurring maintenance work.
- Added `wiki/spec/campaign-operations-model.md` as the new architecture-level source of truth for the operating model shift from a consecutive workflow into a campaign operations system: operator as manager, orchestrator as campaign manager, specialists as tactically autonomous workers, campaigns as durable workspaces, work items plus schedules as the main coordination primitives, and shared campaign memory plus agent working memory as the long-lived continuity model.
- Added `wiki/plan/campaign-memory-redesign.md` to capture the next storage-direction decision: keep runtime state small and structured, but move campaign planning and long-lived marketing knowledge toward a flexible file-backed workspace with Markdown-first memory and lightweight campaign metadata instead of deepening the current fixed-artifact JSON model.
- Implemented persistent Telegram polling cursor state so long-poll restarts do not replay the same inbound Bot API update: `telegram_app/polling_runner.py` now restores and saves the last successfully delivered `update_id`, skips already-processed duplicates, and only advances the cursor after successful response delivery; focused polling tests now cover retry, restore, persistence, and duplicate-skip behavior.
- Hardened the local Telegram polling runner for live testing: `telegram_app/polling_runner.py` now retries `deleteWebhook` and `getUpdates` after transient Bot API failures instead of letting the whole poller die, and `tests/test_polling_runner.py` now covers both retry paths.
- Cleaned up `wiki/plan/target-search-refinement.md` so the “shipped” status and remaining-work sections now align with the actual discovery runtime: stale pre-build gap language was rewritten as historical context already closed by the current build.
- Finished the remaining target-search refinement code slices: discovery now persists explicit harvest-versus-validation diagnostics, shortlist communities carry compact `evidence_summary` and `validation_path` evidence, exact live validation now wins ranking ties over broader harvest matches, and operator/downstream shortlist summaries now explain sparse coverage and fallback reliance without passing raw search traces forward.
- Implemented the broader MTProto search fallback from the target-search plan: `CommunityCapabilityImpl.search()` now uses `messages.SearchGlobalRequest` when `harvest` mode gets sparse `contacts.search` results, merges and deduplicates global chats behind the existing bounded flow, preserves per-result `search_source`, and has focused MTProto tests covering both the fallback and exact-mode no-fallback path.
- Implemented the bounded refinement slice from the target-search plan: discovery now runs one deterministic second-pass harvest only when first-pass live coverage is sparse, persists refinement diagnostics including productive families and added candidates, and has focused runtime tests covering the trigger and bounded behavior.
- Expanded the wiki support around target-search refinement: `wiki/plan/target-search-refinement.md` now distinguishes shipped slices from remaining work, `wiki/code-index/discovery-search.md` maps the live discovery-search path and boundaries, and `wiki/code-index/index.md` now points directly at that narrow search map.
- Implemented the next target-search refinement slice in code: `CommunityCapability.search()` now accepts explicit `mode` and `limit` parameters, the MTProto and stub community capabilities preserve those search semantics in their results, discovery brief searches now run in `harvest` mode with a larger limit, and shortlist enrichment now tries the already-harvested live candidate pool before issuing fresh exact-match requeries.
- Implemented the first code slice of target-search refinement in discovery: brief search generation now uses a broader deterministic query budget with related-phrase and city-hub variants, the LLM now receives a compact `community_search_summary` instead of raw per-query search dumps, and persisted shortlist artifacts now carry `search_diagnostics` with per-query coverage metrics while downstream strategy/account-planning prompt payloads continue to exclude those bulky diagnostics.
- Rebuilt `wiki/plan/target-search-refinement.md` into a tighter implementation plan aligned to the current discovery runtime: it now captures the shipped baseline, the remaining search-coverage gaps, phased file-level work, acceptance criteria, non-goals, and a recommended delivery order for broader live Telegram discovery without prompt-context bloat.
- Tightened `wiki/plan/target-search-refinement.md` to make the build-critical decisions explicit without expanding scope: compact discovery prompt summaries must land before broader query execution, prompt-contract updates are part of the implementation track, public `verification_state` labels stay stable for this build, and new lookup references should be added with temporary backward compatibility for downstream consumers.
- Added token-efficiency guardrails to `wiki/plan/target-search-refinement.md`, documenting the main prompt-spend risks in the current discovery/search flow and the design constraints needed to keep broader search logic from bloating model context.
- Added a bounded query-refinement loop to `wiki/plan/target-search-refinement.md`: after the deterministic first pass, discovery may run one tightly constrained second pass based only on query families that already produced useful live candidates.
- Added a plain-language explanation section to `wiki/plan/target-search-refinement.md` so the search-expansion strategy, staged validation model, evidence-preservation rationale, and accuracy guardrails are understandable without reading the implementation details alone.
- Expanded `wiki/plan/target-search-refinement.md` with MTProto-specific search constraints and a staged solution shape, including deterministic query-family expansion, hybrid `contacts.search` plus `messages.searchGlobal` harvesting, stable lookup-reference preservation, and first-class search diagnostics.
- Added `wiki/plan/target-search-refinement.md` to capture discovery-search issues found in live review, including brittle brief query generation, shallow Telethon search depth, validation-first search flow, and permissive approximate matching of training-seeded candidates to live Telegram entities.
- Removed hard runtime approval gates from the discovery, strategy, and account-planning flow. The runtime now keeps those checkpoints conversational: specialists still ask the operator whether to continue or revise, but the session no longer blocks on persisted `waiting_for_approval` state for planning artifacts.
